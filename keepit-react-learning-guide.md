# 📚 לומדים React דרך KeepIt — מדריך הלמידה המלא

מסמך זה מסכם את כל מושגי React והעקרונות שנלמדו תוך כדי בניית KeepIt — אפליקציה אישית לניהול שוברים. כל מושג מקושר לקובץ אמיתי בפרויקט, כך שאפשר לקרוא את ההסבר ומיד לראות אותו חי בקוד.

---

## חלק א' — הארכיטקטורה: מי עושה מה

### מבנה הפרויקט

```
keep-it/
├── index.html                  # הדף היחיד; React משתלט על div#root
├── vite.config.js              # הגדרות כלי הבנייה (Vite + Tailwind)
├── package.json                # רשימת התלויות והסקריפטים
└── src/
    ├── main.jsx                # נקודת הכניסה — מחבר את React ל-DOM
    ├── App.jsx                 # הקומפוננטה הראשית: דשבורד + ניתוב
    ├── index.css               # טוקני עיצוב (Tailwind v4) + סגנון התלוש
    ├── db.js                   # שכבת הנתונים — כל הגישה ל-IndexedDB
    ├── utils/
    │   └── expiry.js           # חישובי תוקף — פונקציות טהורות
    └── components/
        ├── VoucherCard.jsx     # כרטיס שובר בודד
        ├── AddVoucherForm.jsx  # טופס יצירה/עריכה
        ├── EmptyState.jsx      # מסך ריק מזמין
        ├── BarcodeScanner.jsx  # סריקה במצלמה (ZXing)
        └── BarcodeModal.jsx    # הצגת ברקוד גרפי (JsBarcode)
```

### עקרון #1: הפרדת שכבות

הקוד מחולק לשלוש שכבות שלא מתערבבות:

| שכבה | קבצים | אחריות |
|---|---|---|
| **נתונים** | `db.js` | כל פעולה מול IndexedDB. אף קומפוננטה לא ניגשת למסד ישירות |
| **לוגיקה** | `utils/expiry.js` | חישובים טהורים בלי React ובלי DOM |
| **תצוגה** | `components/`, `App.jsx` | רק רינדור ואינטראקציה |

**למה זה חשוב:** כשנרצה להחליף את IndexedDB בשרת אמיתי — נשנה רק את `db.js`. כשנרצה לבדוק את לוגיקת התוקף — נבדוק פונקציות פשוטות בלי להריץ UI.

### עקרון #2: פונקציות טהורות (Pure Functions)

ב-`utils/expiry.js` הפונקציות `daysLeft`, `expiryStatus`, `expiryLabel` הן "טהורות": אותו קלט ⬅ תמיד אותו פלט, בלי תלות ב-state, ב-DOM או ברשת. פונקציות כאלה קלות לבדיקה, לשימוש חוזר, ולהבנה. כלל אצבע: כל לוגיקה שאפשר להוציא מקומפוננטה לפונקציה טהורה — כדאי להוציא.

## חלק ב' — שלבים 1–2: יסודות React

### 1. קומפוננטות (Components)

קומפוננטה = פונקציית JavaScript שמחזירה JSX (תחביר דמוי-HTML). האפליקציה היא עץ של קומפוננטות:

```
App
├── VoucherCard (× כמה שוברים)
├── AddVoucherForm
└── EmptyState
```

📁 **דוגמה בקוד:** כל קובץ ב-`src/components/` הוא קומפוננטה אחת. עיקרון: קובץ = קומפוננטה = אחריות אחת.

### 2. Props — נתונים שזורמים למטה

props הם ה"פרמטרים" של קומפוננטה — נתונים שההורה מעביר לילד. הזרימה היא תמיד **מלמעלה למטה**, והילד לא משנה את ה-props שקיבל.

📁 **דוגמה בקוד:** `App.jsx` מעביר לכל כרטיס את השובר שלו:
```jsx
<VoucherCard key={v.id} voucher={v} onEdit={openEdit} />
```
שים לב: prop יכול להיות גם **פונקציה** (`onEdit`) — ראו "הרמת state" בחלק ד'.

### 3. State — הזיכרון של הקומפוננטה

`useState` נותן לקומפוננטה זיכרון ששורד בין רינדורים. כשה-state משתנה — React מרנדר מחדש אוטומטית.

```jsx
const [screen, setScreen] = useState('dashboard')
```

שלושה כללים:
- משנים **רק** דרך פונקציית ה-set, אף פעם לא ישירות
- כשהערך החדש תלוי בישן — מעבירים פונקציה: `setForm(prev => ({ ...prev, name: value }))`
- state הוא מקומי לקומפוננטה; כשכמה קומפוננטות צריכות אותו — מרימים אותו להורה המשותף

📁 **דוגמה בקוד:** `App.jsx` משתמש ב-`screen` כ"ניתוב" פשוט בין הדשבורד לטופס — בלי ספריית ניתוב. כשהאפליקציה תגדל, נחליף ב-react-router.

### 4. Controlled Inputs — התבנית לטפסים

בטופס "מבוקר", ה-state של React הוא מקור האמת היחיד לערך של כל שדה: ה-input מציג את מה שב-state (`value={form.business}`), וכל הקלדה מעדכנת את ה-state (`onChange={handleChange}`).

📁 **דוגמה בקוד:** `AddVoucherForm.jsx` — כולל טריק שחוסך קוד: אובייקט state אחד לכל הטופס ופונקציית עדכון אחת לכל השדות, לפי `name` של ה-input:

```jsx
function handleChange(e) {
  const { name, value } = e.target
  setForm((prev) => ({ ...prev, [name]: value }))
}
```

### 5. רינדור מותנה ורשימות

- **תנאי:** `{scanning && <BarcodeScanner />}` — הקומפוננטה קיימת רק כשהתנאי אמת. זו הדרך להציג/להסתיר מודלים ומסכים.
- **רשימות:** `{active.map(v => <VoucherCard key={v.id} ... />)}` — כל פריט חייב `key` ייחודי ויציב, שעוזר ל-React לזהות מה השתנה בלי לרנדר הכול מחדש.

📁 **דוגמה בקוד:** `App.jsx` — גם הרשימה, גם הסתרת המומשים, גם המעבר בין מסכים.

### 6. useLiveQuery — נתונים חיים (Dexie)

ה-hook `useLiveQuery` מספריית dexie-react-hooks מאזין למסד הנתונים: כל הוספה/מחיקה/עדכון מרנדרים את הקומפוננטה מחדש אוטומטית. לכן `VoucherCard` יכול לקרוא ל-`deleteVoucher` — והרשימה ב-`App` מתעדכנת "לבד", בלי להעביר אירועים ידנית.

📁 **דוגמה בקוד:** `App.jsx`:
```jsx
const vouchers = useLiveQuery(async () => { ... })
if (!vouchers) return null   // בשליפה הראשונה הערך הוא undefined
```

### 7. IndexedDB + Dexie — נתונים בלי שרת

IndexedDB הוא מסד נתונים מובנה בדפדפן; Dexie עוטפת אותו ב-API נוח. בהגדרת הסכמה מציינים רק שדות לחיפוש/מיון — שאר השדות נשמרים חופשי:

```js
db.version(1).stores({ vouchers: '++id, business, expiry, redeemed' })
```

`++id` = מזהה אוטומטי עולה. 📁 **דוגמה בקוד:** `db.js`.

## חלק ג' — שלב 3: hooks מתקדמים וסריקת ברקוד

### 8. useRef — ידית אחיזה ל-DOM אמיתי

בדרך כלל ב-React לא נוגעים ב-DOM ישירות. אבל ספריות חיצוניות (ZXing, JsBarcode) צריכות אלמנט אמיתי — `<video>` או `<svg>`. `useRef` נותן "ידית":

```jsx
const videoRef = useRef(null)
// ...
<video ref={videoRef} />
// ומכאן: videoRef.current הוא האלמנט האמיתי
```

בונוס: ref לא גורם לרינדור מחדש כשהוא משתנה — בניגוד ל-state.

📁 **דוגמה בקוד:** `BarcodeScanner.jsx` (וידאו), `BarcodeModal.jsx` (SVG).

### 9. useEffect — תופעות לוואי ו-cleanup

`useEffect` מריץ קוד **אחרי** שהקומפוננטה מופיעה על המסך — המקום לפעולות שנוגעות בעולם שמחוץ ל-React: הפעלת מצלמה, טיימרים, מאזינים, בקשות רשת.

החלק הקריטי: הפונקציה **שמוחזרת** מה-effect היא ה-cleanup, והיא רצה כשהקומפוננטה מוסרת:

```jsx
useEffect(() => {
  // הפעלת המצלמה...
  return () => controls?.stop()   // בלי זה — המצלמה נשארת דולקת!
}, [onDetected])
```

המערך בסוף (`[onDetected]`) הוא רשימת התלויות — ה-effect ירוץ מחדש רק אם אחת מהן השתנתה.

📁 **דוגמה בקוד:** `BarcodeScanner.jsx` — הפעלה וכיבוי של מצלמה. `BarcodeModal.jsx` — ציור הברקוד מחדש כשהמספר משתנה.

### 10. lazy + Suspense — פיצול קוד (Code Splitting)

ספריית ZXing שוקלת ~0.5MB. במקום שכל מי שפותח את האפליקציה יוריד אותה, מפצלים:

```jsx
const BarcodeScanner = lazy(() => import('./BarcodeScanner'))
// ...
<Suspense fallback={<div>טוען סורק...</div>}>
  <BarcodeScanner ... />
</Suspense>
```

`lazy` הופך את הייבוא לקובץ נפרד שנטען רק בשימוש הראשון; `Suspense` מגדיר מה מציגים בזמן ההורדה. **התוצאה אצלנו:** הבאנדל הראשי ירד מ-770KB ל-324KB.

📁 **דוגמה בקוד:** `AddVoucherForm.jsx`.

### 11. ידע דפדפן: Secure Context

גישה למצלמה (`getUserMedia`) מותרת רק ב-**HTTPS או localhost**. לכן הסריקה עובדת ב-`npm run dev` במחשב, אבל תיחסם מהטלפון ברשת ביתית על HTTP. פתרון: פריסה ל-Vercel/Netlify (HTTPS חינם).

## חלק ד' — שלב העריכה: תבניות מתקדמות

### 12. הרמת State למעלה (Lifting State Up)

**הבעיה:** הכרטיס יודע איזה שובר רוצים לערוך; הטופס צריך את הנתונים. איך שתי קומפוננטות "אחיות" מדברות?

**הפתרון:** ה-state עולה להורה המשותף. `App` מחזיק `editing`, מעביר לכרטיס פונקציה (`onEdit`), והכרטיס רק "מדווח למעלה":

```
App  ← מחזיק editing
 ├── VoucherCard  ← קורא ל-onEdit(voucher)
 └── AddVoucherForm  ← מקבל voucher={editing}
```

זו התבנית המרכזית לתקשורת בין קומפוננטות ב-React: **נתונים למטה דרך props, אירועים למעלה דרך פונקציות-props**.

📁 **דוגמה בקוד:** `App.jsx` (openEdit / closeForm), `VoucherCard.jsx` (הכפתור "ערוך").

### 13. שימוש חוזר בקומפוננטה — טופס אחד, שני מצבים

במקום `AddForm` ו-`EditForm` נפרדים (וכפילות קוד), טופס אחד עם prop אופציונלי:

```jsx
export default function AddVoucherForm({ onClose, voucher = null }) {
  const isEdit = Boolean(voucher)
  // isEdit קובע: כותרת, ערכי התחלה, ופעולת השמירה (add או update)
}
```

📁 **דוגמה בקוד:** `AddVoucherForm.jsx`.

### 14. useState עם פונקציה — Lazy Initializer

```jsx
const [form, setForm] = useState(() => isEdit ? {...מהשובר} : {...ריק})
```

כשמעבירים ל-`useState` **פונקציה** במקום ערך — היא רצה פעם אחת בלבד, ברינדור הראשון. בלי זה, בניית אובייקט האתחול הייתה מתבצעת מחדש בכל הקלדה (מבוזבז), ובחישובים כבדים זה קריטי.

### 15. key כמנגנון איפוס

הטריק האלגנטי של השלב:

```jsx
<AddVoucherForm key={editing?.id ?? 'new'} voucher={editing} onClose={closeForm} />
```

כש-`key` משתנה, React **הורס** את הקומפוננטה ובונה חדשה עם state נקי. ככה מעבר מעריכת שובר א' לשובר ב' פותח טופס עם הערכים הנכונים — בלי שום קוד איפוס ידני. זו הדרך הרשמית (מהתיעוד של React) לאפס state כשה"זהות" של הנתונים מתחלפת.

📁 **דוגמה בקוד:** `App.jsx`.

## חלק ו' — PWA ותזכורות תוקף

### 16. PWA — Web App Manifest + Service Worker

`vite-plugin-pwa` מוסיף לבנייה שני דברים: **manifest** (שם, אייקונים, צבעים — מאפשר "הוסף למסך הבית") ו-**service worker** (precache של כל הקבצים, כדי שהאפליקציה תעבוד גם אופליין). שני אלה מוגדרים כתצורה בתוך `vite.config.js` — אין קובץ manifest.json נפרד לכתוב ידנית.

📁 **דוגמה בקוד:** `vite.config.js` (בלוק `VitePWA`), `index.html` (תגית `apple-touch-icon` — iOS לא קורא אייקונים מה-manifest).

⚠️ **מלכודת:** ה-service worker פעיל רק ב-`npm run build && npm run preview` — לא ב-`npm run dev`.

### 17. Notifications API — תופעת לוואי תלוית-הרשאה

`Notification.requestPermission()` היא קריאה שחייבת לרוץ בתגובה לפעולת משתמש (לחיצה על כפתור) — דפדפנים חוסמים בקשת הרשאה אוטומטית. אחרי אישור, `new Notification(title, options)` יוצר התראה מיידית.

📁 **דוגמה בקוד:** `App.jsx` — `enableReminders()` מבקש הרשאה; ה-`useEffect` שתלוי ב-`vouchers` בודק בכל עדכון מסד אילו שוברים עומדים לפוג ושולח התראה **פעם אחת ביום** לכל שובר, בעזרת שדה `notifiedOn` שנשמר על הרשומה עצמה (`markNotified` ב-`db.js`) — זה מונע הצפה של התראות באותו יום.

⚠️ **מגבלה מובנית:** זה עובד רק כשהאפליקציה פתוחה בדפדפן. התראות "אמיתיות" כשהאפליקציה סגורה (Push API) דורשות שרת שיישלח אותן — וזה סותר את עקרון "אין backend" של הפרויקט.

## חלק ז' — לאן ממשיכים

מה שעוד מחכה בפרויקט, וכל אחד מהם ילמד משהו חדש:

1. **react-router** — ניתוב אמיתי עם כתובות URL במקום ה-state הפשוט
2. **צילום השובר** — שמירת תמונות כ-Blob ב-IndexedDB
3. **ייצוא/ייבוא נתונים** — גיבוי JSON

טיפ אחרון ללמידה: אל תסתפק בקריאה — **שבור דברים בכוונה**. מחק את ה-cleanup בסורק ותראה שהמצלמה נשארת דולקת. הסר את ה-`key` מהטופס ותראה את באג הערכים הישנים. ככה המושגים באמת נצרבים.
